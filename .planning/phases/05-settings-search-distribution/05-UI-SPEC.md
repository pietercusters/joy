# Phase 5 -- UI Design Contract

> Visual and interaction contract for the Settings, Search & Distribution phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** joy is a Python TUI built with Textual 8.x. Design tokens map to Textual CSS variables, not web CSS. All spacing values are expressed in terminal cell units (1 cell = 1 character width or height), not pixels. Colors reference Textual's built-in theme system.

| Property | Value |

## Spacing Scale

Textual CSS uses integer cell units. All spacing in this contract is expressed in **terminal cells** (characters), not pixels.

| Token | Value | Usage |
|-------|-------|-------|
| none | 0 | No gap between dot indicator and icon (existing) |
| xs | 1 | Inline padding (`padding: 0 1` on ObjectRow, existing); horizontal padding inside modal containers; gap between field label and input field |
| sm | 2 | Gap between icon and label, label and value (existing); `padding: 1 2` inside SettingsModal container (1 vertical, 2 horizontal) |
| md | 3 | Not used in this phase |
| lg | 4 | Not used in this phase |

Exceptions: none. Phase 5 modals and inline widgets use the same spacing vocabulary established in Phases 2-4.

**Source:** Phase 4 UI-SPEC spacing scale; existing modal `padding: 1 2` pattern.

## Typography

Textual renders in the terminal's monospace font. No font-size or font-weight choices in the traditional sense. Textual CSS supports `text-style` (bold, italic, underline) and `color` for emphasis.

| Role | Style | Usage |
|------|-------|-------|
| Body | normal | Settings field values, filter input text, project names |
| Label | normal + `$text-muted` color | Settings field labels ("IDE", "Editor", "Obsidian Vault Path", "Terminal", "Default Open Kinds") |
| Modal title | bold + `$text` color | "Settings" title at top of SettingsModal |
| Hint text | normal + `$text-muted` color | "Tab to navigate, Enter / Save Settings to save, Escape to cancel" in SettingsModal; filter input placeholder text |
| Button label | normal | "Save Settings" button text in SettingsModal |
| Toast message | normal (Textual notify default) | "Settings saved" feedback |
| Version output | normal (stdout print) | `joy 0.1.0` on `--version` flag |

**Source:** Phase 4 UI-SPEC typography; existing modal pattern (NameInputModal, ConfirmationModal).

## Color

Textual uses a built-in theme system with CSS variables. The app uses Textual's default dark theme with no custom CSS overrides (Phase 2 decision D-11).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `$surface` (Textual default dark bg) | App background, pane backgrounds, modal overlay dimmed background, SettingsModal background |
| Secondary (30%) | `$panel` / `$surface` | SettingsModal container background, input field backgrounds, filter input background |
| Accent (10%) | `$accent` | Highlighted row background (existing), focused input border in SettingsModal fields, focused filter input border, Save Settings button `variant="primary"` |
| Text | `$text` (white/light in dark theme) | Modal title text, input text, button labels, filter input text |
| Muted | `$text-muted` | Field labels in SettingsModal, placeholder text in filter input, hint text in modals |
| Error | `severity="error"` on `app.notify()` | Failed save toasts, edge case errors |

**Accent reserved for:**
- Highlighted row background (existing Phase 2 behavior)
- Focused input field border in SettingsModal
- Focused filter input border
- Save Settings button primary variant styling
- Selected items in SelectionList (default_open_kinds checklist)

**No destructive color in this phase:** Phase 5 has no destructive actions. No `$error` border needed on any modal.

**Source:** Phase 4 UI-SPEC color contract; 05-CONTEXT.md.

## Settings Modal Design Contract

### SettingsModal (ModalScreen overlay)

**Trigger:** `s` key (global, bound on `JoyApp` with `priority=True`)
**Purpose:** View and edit the 5 global Config fields
**Dismiss behavior:** Escape cancels without saving (returns `None`); Save Settings button persists changes (returns updated `Config`)

| Property | Value |
|----------|-------|
| Screen type | `ModalScreen[Config \| None]` subclass |
| Invocation | `self.push_screen(SettingsModal(self._config), callback)` |
| Overlay | Textual default ModalScreen overlay (semi-transparent dimmed background) |
| Alignment | `align: center middle` on the SettingsModal |
| Width | `width: 70` cells (wider than Phase 4 modals to accommodate field labels) |
| Height | `height: auto` with `max-height: 80vh` on inner container |
| Overflow | `overflow: auto` on inner Vertical container for small terminals |
| Background | `background: $surface` on inner Vertical |
| Border | `border: thick $background 80%` (standard non-destructive modal border) |
| Padding | `padding: 1 2` inside the modal container |

**Source:** 05-CONTEXT.md D-01, D-02; 05-RESEARCH.md Pattern 1.

### Settings Modal Navigation

| Key | Action |
|-----|--------|
| `Tab` | Advance focus to next focusable widget (Input -> Input -> ... -> SelectionList -> Button) |
| `Shift+Tab` | Return focus to previous focusable widget |
| `Escape` | Dismiss modal, return `None` (no save) |
| `Enter` on Save Settings button | Collect all field values, dismiss modal, return updated `Config` |
| `Space` inside SelectionList | Toggle the focused selection item |

Tab navigation is automatic via Textual's built-in `Screen.BINDINGS` (`tab -> app.focus_next`, `shift+tab -> app.focus_previous`). No manual implementation needed.

**Initial focus:** `#field-ide` (first Input field) receives focus on mount.

**Source:** 05-CONTEXT.md D-04; 05-RESEARCH.md Tab navigation verification.

## Inline Filter Design Contract

### Filter Input (inline widget in ProjectList)

**Trigger:** `/` key (pane-scoped, bound on `JoyListView` with `show=True`)
**Purpose:** Real-time substring filter on the project list
**Exit:** Escape unmounts the input and restores the full list

| Property | Value |
|----------|-------|
| Widget | `Input(placeholder="Filter projects...", id="filter-input")` |
| Mount location | Inside `ProjectList` (the parent Widget), mounted `before=` the `JoyListView` |
| Mount method | `parent.mount(filter_input, before=listview_widget)` |
| Remove method | `filter_input.remove()` |
| Height | 1 cell (default Input height) |
| Width | Inherits parent width (fills the left pane) |

**Source:** 05-CONTEXT.md D-06; 05-RESEARCH.md Pattern 2.

### Filter Behavior

| Event | Behavior |
|-------|----------|
| `/` pressed on JoyListView | Mount Input above list, set `_filter_active = True`, focus Input |
| Keystroke in filter Input (`Input.Changed`) | Filter `JoyApp._projects` by case-insensitive substring match on `.name`; call `set_projects(filtered_subset)` |
| Input cleared (empty string) | Restore full list via `set_projects(JoyApp._projects)` |
| Escape pressed while filter active | Remove Input, set `_filter_active = False`, restore full list via `set_projects(JoyApp._projects)`, refocus JoyListView |
| Enter pressed in filter Input | Dismiss filter Input (keep current filtered subset visible), refocus JoyListView |
| `/` pressed while filter already active | No-op (prevent duplicate Input mount) |

**Canonical list protection (D-09):** `JoyApp._projects` is never mutated by filtering. `ProjectList.set_projects()` receives a filtered copy. On filter exit, the canonical list is always restored from `self.app._projects`.

**Escape conflict avoidance:** Escape during filter mode is handled via `on_key` with a `_filter_active` guard on `ProjectList`, calling `event.stop()` to prevent the event from propagating to parent widgets. This avoids conflicting with ModalScreen Escape behavior.

**Source:** 05-CONTEXT.md D-06 through D-10; 05-RESEARCH.md Pitfall 1 and Pitfall 3.

### Filter Visual State

| State | Left Pane Appearance |
|-------|---------------------|
| Filter inactive (default) | JoyListView shows all projects, no Input visible |
| Filter active, query empty | Input visible above list with placeholder "Filter projects..."; full project list shown |
| Filter active, query typed | Input visible with typed text; project list shows matching subset only |
| Filter active, no matches | Input visible with typed text; project list is empty |

## Interaction Contract

### Key Bindings (Phase 5 additions)

| Key | Scope | Action | Condition |
|-----|-------|--------|-----------|
| `s` | `JoyApp` (global -- any pane, `priority=True`) | Open SettingsModal overlay | Always available |
| `/` | `JoyListView` (project list focused, `show=True`) | Enter filter mode: mount inline Input above project list | Not already in filter mode |

**Source:** 05-CONTEXT.md D-05, D-10; 05-RESEARCH.md Pattern 4.

### Flow 1: Edit Settings (`s`)

1. User presses `s` from any pane
2. `JoyApp.action_settings()` pushes `SettingsModal(self._config)` with callback
3. SettingsModal appears centered with dimmed background; all 5 fields pre-populated from current `Config`
4. User navigates between fields with Tab/Shift+Tab
5. User edits Input values by typing; toggles SelectionList items with Space
6. **On Escape:** modal dismissed with `None`; no changes saved; return to previous state
7. **On Save Settings button press:** modal collects all field values into a new `Config` object, dismisses with the `Config`
8. Callback: `self._config = config`, call `_save_config_bg()` (persists via `save_config()` in `@work(thread=True)`), show toast `"Settings saved"`

**Source:** 05-CONTEXT.md D-01 through D-05; 05-RESEARCH.md Pattern 1, Pattern 4.

### Flow 2: Filter Projects (`/`)

1. User presses `/` while JoyListView is focused
2. `JoyListView.action_filter()` checks `_filter_active`; if already active, no-op
3. Mounts `Input(placeholder="Filter projects...", id="filter-input")` into `ProjectList` before the JoyListView
4. Sets `_filter_active = True`, focuses the Input
5. As user types, `ProjectList.on_input_changed()` filters `JoyApp._projects` by case-insensitive substring match on `project.name`, calls `set_projects(filtered_list)`
6. **On Escape:** `ProjectList.on_key()` catches Escape when `_filter_active`, calls `_exit_filter_mode()`: removes Input, restores full list from `JoyApp._projects`, refocuses JoyListView
7. **On Enter:** `ProjectList.on_input_submitted()` calls `_exit_filter_mode()` but keeps the current filtered subset visible (does not restore full list); refocuses JoyListView
8. **On clearing all text (backspace to empty):** `on_input_changed` fires with empty string; restores full list

**Source:** 05-CONTEXT.md D-06 through D-10; 05-RESEARCH.md Pattern 2.

### Flow 3: Version Check (`--version`)

1. User runs `joy --version` from shell
2. `main()` in `app.py` checks `sys.argv` **before** creating `JoyApp`
3. If `sys.argv[1] == "--version"`: import `importlib.metadata`, call `importlib.metadata.version("joy")`, print `joy {version}`, return
4. If `PackageNotFoundError`: print `joy unknown`
5. No TUI initialization, no data loading

**Source:** 05-CONTEXT.md D-11; 05-RESEARCH.md Pattern 3.

## Footer Binding Display

Phase 5 adds new bindings to the existing context-sensitive footer.

### Detail pane focused

Existing (Phases 2-4): `Up/k | Down/j | o: Open | O: Open All | Space: Toggle | a: Add | e: Edit | d: Delete | n: New | Esc: Back`

Phase 5 additions: `s: Settings` (global, visible from any pane)

Full footer: `Detail | Up/k | Down/j | o: Open | O: Open All | Space: Toggle | a: Add | e: Edit | d: Delete | n: New | s: Settings | Esc: Back`

### Project list focused

Existing (Phases 2-4): `Up/k | Down/j | Enter: Open | O: Open All | n: New | D: Delete | q: Quit`

Phase 5 additions: `/ : Filter` (pane-scoped, visible when list focused), `s: Settings` (global)

Full footer: `Projects | Up/k | Down/j | Enter: Open | O: Open All | n: New | D: Delete | /: Filter | s: Settings | q: Quit`

### Filter mode active (Input focused)

When the filter Input is focused, the footer shows Input-level bindings. Textual's Footer widget automatically reflects the focused widget's bindings. The filter Input has no custom BINDINGS, so the footer may show minimal hints. The Escape key handler is implicit via `on_key`.

**Note:** Footer will truncate if terminal is narrow. Textual's built-in Footer widget handles this. `s` binding uses `priority=True` and `show=True`. `/` binding uses `show=True`.

**Source:** Phase 4 UI-SPEC footer section; 05-CONTEXT.md D-05, D-10; 05-RESEARCH.md Open Question 3.
