# Phase 4 -- UI Design Contract

> Visual and interaction contract for the CRUD phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** joy is a Python TUI built with Textual 8.x. Design tokens map to Textual CSS variables, not web CSS. All spacing values are expressed in terminal cell units (1 cell = 1 character width or height), not pixels. Colors reference Textual's built-in theme system.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Python TUI -- shadcn not applicable) |
| Preset | not applicable |

## Spacing Scale

Textual CSS uses integer cell units. All spacing in this contract is expressed in **terminal cells** (characters), not pixels.

| Token | Value | Usage |
|-------|-------|-------|
| none | 0 | No gap between dot indicator and icon (existing) |
| xs | 1 | Inline padding (`padding: 0 1` on ObjectRow, existing); horizontal padding inside modal containers |
| sm | 2 | Gap between icon and label, label and value (existing pattern); vertical gap between modal widgets |
| md | 3 | Vertical padding above/below modal content area |
| lg | 4 | Not used in this phase |

Exceptions: none. Phase 4 modals use the same spacing vocabulary established in Phases 2-3.

**Source:** Phase 3 UI-SPEC spacing scale; existing ObjectRow `_render_text` uses 2-space gaps.

## Typography

Textual renders in the terminal's monospace font. No font-size or font-weight choices in the traditional sense. Textual CSS supports `text-style` (bold, italic, underline) and `color` for emphasis.

| Role | Style | Usage |
|------|-------|-------|
| Body | normal | Object row values, project names, modal input values |
| Label | normal | Preset kind labels in object rows, form field labels in modals |
| Group header | bold + `$text-muted` color | Section headers in detail pane (existing Phase 2 pattern) |
| Modal title | bold + `$text` color | Title text at top of each modal dialog (e.g., "New Project", "Add Object") |
| Prompt text | normal + `$text` color | Confirmation dialog question text |
| Item name in confirmation | bold | The project name or object value shown in delete confirmations for emphasis |
| Toast message | normal (Textual notify default) | CRUD feedback messages |

**Source:** Phase 3 UI-SPEC typography; Phase 2 codebase GroupHeader pattern.

## Color

Textual uses a built-in theme system with CSS variables. The app uses Textual's default dark theme with no custom CSS overrides (Phase 2 decision D-11).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `$surface` (Textual default dark bg) | App background, pane backgrounds, modal overlay dimmed background |
| Secondary (30%) | `$panel` / `$surface` | Modal dialog background, input field backgrounds |
| Accent (10%) | `$accent` | Highlighted row background (existing), focused input border, highlighted preset in type-to-filter picker |
| Text | `$text` (white/light in dark theme) | Modal title text, input text, button labels |
| Muted | `$text-muted` | Placeholder text in input fields, preset picker hint text, secondary labels |
| Error | `severity="error"` on `app.notify()` | Failed operation toasts, duplicate project name error |
| Destructive | `$error` (Textual built-in red) | Delete confirmation dialog border accent, destructive action emphasis text |

**Accent reserved for:**
- Highlighted row background (existing Phase 2 behavior)
- Focused input field border in modal forms
- Currently highlighted preset in the type-to-filter picker

**Destructive color reserved for:**
- Delete confirmation modal border (`border: thick $error`)
- The word "Delete" in confirmation dialog prompt text

**Source:** Phase 3 UI-SPEC color contract; 04-CONTEXT.md D-14, D-15.

## Modal Overlay Design Contract

Phase 4 introduces four modal screen types. All share a common visual foundation.

### Common Modal Properties

| Property | Value |
|----------|-------|
| Screen type | `ModalScreen` subclass (blocks app-level key bindings) |
| Invocation | `self.app.push_screen(ModalInstance(), callback)` |
| Overlay | Textual default ModalScreen overlay (semi-transparent dimmed background) |
| Alignment | `align: center middle` on the ModalScreen |
| Dismiss | `self.dismiss(result)` returns data to callback; Escape always cancels via `self.dismiss(None)` |
| Width | `width: 60` cells (fixed -- sufficient for all form content) |
| Background | `background: $surface` |
| Border | `border: thick $background 80%` for input modals; `border: thick $error` for delete confirmations |
| Padding | `padding: 1 2` inside the modal container |

**Source:** 04-CONTEXT.md D-07, D-14; Textual ModalScreen API.

### Modal 1: Name Input (Project Creation -- Step 1)

**Trigger:** `n` key (global, bound on `JoyApp`)
**Purpose:** Capture new project name

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"New Project"` |
| Input | `Input` with placeholder | placeholder: `"Project name"` |
| Hint | `Static` (muted) | `"Enter to create, Escape to cancel"` |

**Return type:** `ModalScreen[str | None]` -- returns the project name string on Enter, `None` on Escape.

**Validation:**
- Empty string: do not dismiss; show inline error via `app.notify("Project name cannot be empty", severity="error")`
- Duplicate name: do not dismiss; show `app.notify("Project '{name}' already exists", severity="error")`

**Height:** `height: auto` (content-sized, approximately 5 rows)

**Source:** 04-CONTEXT.md D-01, D-02, D-04; Claude's discretion for duplicate name handling.

### Modal 2: Preset Picker (Add Object -- Step 1)

**Trigger:** `a` key (bound on `ProjectDetail`) or auto-opens after project creation name modal
**Purpose:** Select a preset kind for the new object via type-to-filter

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"Add Object"` |
| Filter input | `Input` with placeholder | placeholder: `"Type to filter..."` |
| Preset list | `ListView` with `ListItem(Label(...))` | All 9 PresetKind values, filtered in real-time |
| Hint | `Static` (muted) | `"j/k to navigate, Enter to select, Escape to cancel"` |

**Preset display format in list:**
Each `ListItem` shows: `{icon}  {preset_name}` where icon is the Nerd Font glyph from `PRESET_ICONS`.

Example filtered view (user typed "br"):
```
  branch
```

Example full view (no filter):
```
  worktree
  branch
  mr
  ticket
  thread
  file
  note
  agents
  url
```

**Filtering behavior:**
- Case-insensitive substring match on preset kind name
- Empty filter shows all 9 presets in GROUP_ORDER
- No matches: list is empty; Enter does nothing; user must clear filter or press Escape
- j/k or arrow keys navigate the filtered list
- Enter selects the highlighted preset

**Return type:** `ModalScreen[PresetKind | None]` -- returns selected PresetKind on Enter, `None` on Escape.

**Height:** `height: auto` (content-sized, max approximately 14 rows: title + input + 9 items + hint + padding)

**Source:** 04-CONTEXT.md D-05, D-06.

### Modal 3: Value Input (Add Object -- Step 2 / Edit Object)

**Trigger:** After preset selection in add flow, or `e` key on highlighted object (bound on `ProjectDetail`)
**Purpose:** Capture or edit the object value

**Add mode:**

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"Add {preset_name}"` (e.g., "Add branch") |
| Input | `Input` with placeholder | placeholder: `"Enter value"` |
| Hint | `Static` (muted) | `"Enter to add, Escape to cancel"` |

**Edit mode:**

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"Edit {preset_name}"` (e.g., "Edit branch") |
| Input | `Input` pre-populated | value: current `ObjectItem.value` |
| Hint | `Static` (muted) | `"Enter to save, Escape to cancel"` |

**Validation:**
- Empty string: do not dismiss; show `app.notify("Value cannot be empty", severity="error")`

**Return type:** `ModalScreen[str | None]` -- returns value string on Enter, `None` on Escape.

**Height:** `height: auto` (content-sized, approximately 5 rows)

**Source:** 04-CONTEXT.md D-06, D-08.

### Modal 4: Confirmation Dialog (Delete Object / Delete Project)

**Trigger:** `d` on highlighted object (bound on `ProjectDetail`) or `delete`/`D` on project list (bound on `JoyListView`)
**Purpose:** Confirm destructive action before execution

**Delete object:**

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"Delete Object"` |
| Prompt | `Static` | `"Delete {kind} '{value}'?"` where value is truncated at 40 chars |
| Hint | `Static` (muted) | `"Enter to delete, Escape to cancel"` |

**Delete project:**

| Element | Widget | Content |
|---------|--------|---------|
| Title | `Static` (bold) | `"Delete Project"` |
| Prompt | `Static` | `"Delete project '{name}'? This will remove it and all its objects."` |
| Hint | `Static` (muted) | `"Enter to delete, Escape to cancel"` |

**Visual distinction for destructive modals:**
- Border: `border: thick $error` (red border instead of standard border)
- The word "Delete" in the title is styled with `$error` color

**Return type:** `ModalScreen[bool]` -- returns `True` on Enter, `False` on Escape.

**Height:** `height: auto` (content-sized, approximately 5 rows)

**Source:** 04-CONTEXT.md D-10, D-11, D-12, D-13, D-14, D-15.

## Interaction Contract

### Key Bindings

| Key | Scope | Action | Condition |
|-----|-------|--------|-----------|
| `n` | `JoyApp` (global -- any pane) | Open project creation flow (name modal then add-object loop) | Always available |
| `a` | `ProjectDetail` (detail pane focused) | Open add-object flow (preset picker then value input, loops until Escape) | Project must be displayed |
| `e` | `ProjectDetail` (detail pane focused) | Open edit modal for highlighted object | Object must be highlighted |
| `d` | `ProjectDetail` (detail pane focused) | Open delete confirmation for highlighted object | Object must be highlighted |
| `delete` or `D` | `JoyListView` (project list focused) | Open delete confirmation for highlighted project | Project must be highlighted |

**Source:** 04-CONTEXT.md D-01, D-05, D-08, D-09, D-10, D-11, D-12.

### Flow 1: Create New Project (`n`)

1. User presses `n` from any pane
2. `JoyApp.action_new_project()` pushes `NameInputModal()` with callback
3. Name modal appears centered; user types project name
4. **On Escape:** modal dismissed with `None`; no-op, return to previous state
5. **On Enter with valid name:** modal dismissed with name string
6. Callback: create `Project(name=name, objects=[])`, append to `JoyApp._projects`, persist via `save_projects()` in `@work(thread=True)`, refresh project list via `ProjectList.set_projects()`, select the new project
7. Immediately push `PresetPickerModal()` to start the add-object loop
8. After preset selection, push `ValueInputModal(preset_kind)` to capture value
9. **On Enter with valid value:** create `ObjectItem(kind=preset_kind, value=value)`, append to project.objects, persist, refresh detail pane, then loop back to step 7 (push PresetPickerModal again)
10. **On Escape at any point in the loop:** exit loop; user lands in detail pane of new project

**Source:** 04-CONTEXT.md D-01, D-02, D-03, D-04.

### Flow 2: Add Object (`a`)

1. User presses `a` in the detail pane
2. `ProjectDetail.action_add_object()` pushes `PresetPickerModal()` with callback
3. Preset picker appears; user types to filter, navigates with j/k, selects with Enter
4. **On Escape:** modal dismissed with `None`; no-op
5. **On Enter:** callback receives `PresetKind`; push `ValueInputModal(preset_kind)` with callback
6. **On Escape from value input:** no-op
7. **On Enter with valid value:** create `ObjectItem(kind=preset_kind, value=value)`, append to project.objects, persist, refresh detail pane
8. Loop: push PresetPickerModal again (same as step 2)
9. **On Escape from preset picker:** exit loop; user returns to detail pane

**Source:** 04-CONTEXT.md D-05, D-06, D-07.

### Flow 3: Edit Object (`e`)

1. User presses `e` in the detail pane
2. `ProjectDetail.action_edit_object()` reads `self.highlighted_object`
3. If `None`: show `app.notify("No object selected", severity="error")`; return
4. Push `ValueInputModal(item.kind, existing_value=item.value)` with callback
5. **On Escape:** no-op
6. **On Enter with valid value:** update `item.value = new_value`, persist via `save_projects()` in `@work(thread=True)`, refresh the specific `ObjectRow` via `_render_project()` or `row.refresh_indicator()`-style update

**Source:** 04-CONTEXT.md D-08, D-09.

### Flow 4: Delete Object (`d`)

1. User presses `d` in the detail pane
2. `ProjectDetail.action_delete_object()` reads `self.highlighted_object`
3. If `None`: show `app.notify("No object selected", severity="error")`; return
4. Push `ConfirmationModal(title="Delete Object", prompt="Delete {kind} '{value}'?")` with callback
5. **On Escape:** dismissed with `False`; no-op
6. **On Enter:** dismissed with `True`; remove the object from `project.objects`, persist, refresh detail pane
7. Cursor position: move to previous row (index - 1) or stay at same index if deleting last item; if list becomes empty, cursor = -1
8. Show toast: `"Deleted: {kind} '{value_truncated}'"`

**Source:** 04-CONTEXT.md D-10, D-11.

### Flow 5: Delete Project (`delete` or `D`)

1. User presses `delete` or `D` in the project list
2. Determine highlighted project from `JoyListView` index + `ProjectList._projects`
3. If no project highlighted: no-op
4. Push `ConfirmationModal(title="Delete Project", prompt="Delete project '{name}'? This will remove it and all its objects.")` with callback
5. **On Escape:** dismissed with `False`; no-op
6. **On Enter:** dismissed with `True`; remove project from `JoyApp._projects`, persist, refresh project list
7. Selection after delete: select next project if available, else previous; if list becomes empty, clear detail pane
8. Show toast: `"Deleted project: '{name}'"`

**Source:** 04-CONTEXT.md D-12, D-13.

## Copywriting Contract

### Toast Messages (CRUD feedback)

| Action | Success Message | Example |
|--------|----------------|---------|
| Create project | `Created project: '{name}'` | `Created project: 'joy'` |
| Add object | `Added: {kind} '{value_truncated}'` | `Added: branch 'feature/crud-modals'` |
| Edit object | `Updated: {kind} '{value_truncated}'` | `Updated: branch 'feature/crud-final'` |
| Delete object | `Deleted: {kind} '{value_truncated}'` | `Deleted: branch 'feature/old-branch'` |
| Delete project | `Deleted project: '{name}'` | `Deleted project: 'old-project'` |

**Value truncation rule:** Same as Phase 3 -- if value exceeds 40 characters, truncate at character 37 + `...`. Display `item.label` if non-empty; otherwise `item.value`.

### Error Messages

| Condition | Message | Severity |
|-----------|---------|----------|
| No object highlighted when `e` or `d` pressed | `No object selected` | `error` |
| Empty project name submitted | `Project name cannot be empty` | `error` |
| Duplicate project name submitted | `Project '{name}' already exists` | `error` |
| Empty value submitted in add/edit form | `Value cannot be empty` | `error` |
| Persistence failure (save_projects raises) | `Failed to save changes` | `error` |

### Empty States

| Condition | Behavior |
|-----------|----------|
| No projects exist (list is empty) | `n` still works to create first project; detail pane shows nothing |
| Project has no objects | `a` works to add first object; `e` and `d` show "No object selected" error toast |
| Preset filter matches nothing | Empty list in picker; Enter does nothing; user must clear filter or Escape |

### Modal Copy Reference

| Modal | Title | Hint Line |
|-------|-------|-----------|
| Name input (create project) | `New Project` | `Enter to create, Escape to cancel` |
| Preset picker (add object) | `Add Object` | `j/k to navigate, Enter to select, Escape to cancel` |
| Value input (add) | `Add {kind}` | `Enter to add, Escape to cancel` |
| Value input (edit) | `Edit {kind}` | `Enter to save, Escape to cancel` |
| Confirm delete object | `Delete Object` | `Enter to delete, Escape to cancel` |
| Confirm delete project | `Delete Project` | `Enter to delete, Escape to cancel` |

### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| Delete object (`d`) | Modal confirmation: "Delete {kind} '{value}'?" -- Enter confirms, Escape cancels |
| Delete project (`delete`/`D`) | Modal confirmation: "Delete project '{name}'? This will remove it and all its objects." -- Enter confirms, Escape cancels |

**Source:** 04-CONTEXT.md D-10, D-12, D-14, D-15.

## Footer Binding Display

Phase 4 adds new bindings to the existing context-sensitive footer.

### Detail pane focused

Existing (Phases 2-3): `Up/k: Up | Down/j: Down | o: Open | O: Open All | Space: Toggle | Escape: Back`

Phase 4 additions: `a: Add | e: Edit | d: Delete | n: New Project`

Full footer: `Detail | Up/k | Down/j | o: Open | O: Open All | Space: Toggle | a: Add | e: Edit | d: Delete | n: New | Esc: Back`

### Project list focused

Existing (Phases 2-3): `Up/k: Up | Down/j: Down | Enter: Open | O: Open All | q: Quit`

Phase 4 additions: `n: New Project | delete/D: Delete Project`

Full footer: `Projects | Up/k | Down/j | Enter: Open | O: Open All | n: New | D: Delete | q: Quit`

**Note:** Footer will truncate if terminal is narrow. Textual's built-in Footer widget handles this. Priority bindings (`n`, `a`, `d`, `e`) should use `show=True`; navigation bindings (`j`, `k`) can use `show=False` since arrow keys are discoverable.

**Source:** Phase 3 UI-SPEC footer section; 04-CONTEXT.md D-01, D-09, D-12.

## Persistence Contract

All mutations follow the established Phase 3 pattern.

| Operation | Mutation Target | Persist Method | Thread Pattern |
|-----------|-----------------|----------------|----------------|
| Create project | Append to `JoyApp._projects` | `save_projects(self._projects)` | `@work(thread=True)` |
| Add object | Append to `project.objects` | `save_projects(self._projects)` | `@work(thread=True)` |
| Edit object | Update `item.value` in-place | `save_projects(self._projects)` | `@work(thread=True)` |
| Delete object | Remove from `project.objects` | `save_projects(self._projects)` | `@work(thread=True)` |
| Delete project | Remove from `JoyApp._projects` | `save_projects(self._projects)` | `@work(thread=True)` |

**Source:** 04-CONTEXT.md D-16; established in Phase 3 via `_save_toggle()` pattern.
